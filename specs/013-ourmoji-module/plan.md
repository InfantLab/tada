# Implementation Plan: Ourmoji Module

**Branch**: `013-ourmoji-module` | **Date**: 2026-04-03 | **Spec**: `/specs/013-ourmoji-module/spec.md`
**Input**: Feature specification from `/specs/013-ourmoji-module/spec.md`

## Summary

Deliver the Ourmoji module as a feature-flagged capability with two core slices: (1) daily Ourmoji ingestion/display and (2) dream experiment orchestration (nightly role assignment, morning submission/guess/reveal, and post-run analytics). The implementation reuses existing Ta-Da primitives (entry type registry, voice recorder, scheduler sweep patterns, Drizzle + SQLite, and notification services) while introducing dedicated experiment persistence and idempotent scheduling.

## Technical Context

**Language/Version**: TypeScript 5.9.3, Vue 3 + Nuxt 4.4.2 (Nitro server)
**Primary Dependencies**: Nuxt/Nitro, Drizzle ORM, Zod, Vitest, Playwright, Nodemailer, web-push
**Storage**: SQLite (`file:../data/db.sqlite`) via Drizzle (`entries` table + new experiment tables)
**Testing**: Vitest unit/integration (`bun run test:run`), targeted API tests via `@nuxt/test-utils/e2e`, optional Playwright for UX-critical flow
**Target Platform**: Web app (mobile-first PWA + desktop) with Node 20-compatible server runtime
**Project Type**: Single Nuxt web application with server API + client UI
**Performance Goals**:
- Daily Ourmoji read path p95 < 200ms server time
- Morning submit+guess roundtrip p95 < 300ms per call
- Nightly assignment sweep handles all active experiments in < 60s
**Constraints**:
- Preserve Receiver blinding until guess submission
- Enforce one active experiment per participant
- Scheduler must be idempotent and safe across repeated sweeps
- Keep MVP aligned with existing `VoiceRecorder` (no custom wrapper)
**Scale/Scope**:
- Initial cohort: 2 named participants, design supports small-group expansion
- One active experiment per user
- 14+ night runs, 23-choice forced guess space

## Phase 0 Research Plan

Research focuses on concrete, repo-aligned decisions:

1. Entry type registration pattern and feature-flag gating behavior.
2. Nightly scheduler architecture and idempotency constraints.
3. Notification channel choice for MVP and delivery failure behavior.
4. Voice flow reuse strategy for dream capture.
5. Statistical computation strategy for binomial significance.

Outputs are captured in `research.md` with final decisions and alternatives.
